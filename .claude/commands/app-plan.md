# /app-plan — 앱 기획서 생성

$ARGUMENTS

---

## Overview

사용자와 1:1 대화를 통해 앱 기획서(MVP 리포트)를 작성한다. **한 번에 하나의 질문만** 하고, **객관식 우선**으로 진행한다. AI가 스스로 알 수 없는 정보(사업 판단, 타겟, 포지셔닝, 수익 목표 등)를 역질문으로 수집하여 14개 섹션 완성도의 기획서를 만든다.

완성된 기획서는 `/build-app` 커맨드에 바로 넣을 수 있는 형식으로 작성한다.

---

## Iron Laws

1. **한 번에 질문 1개만** — 여러 질문을 한꺼번에 던지지 않는다
2. **객관식 우선** — 가능하면 2~4개 선택지 제시 (자유 입력도 허용)
3. **추측 금지** — 사용자가 답하지 않은 정보는 빈칸으로 두거나 재질문
4. **점진적 검증** — 각 섹션 완성 후 사용자에게 확인 받고 다음으로 진행
5. **YAGNI** — 불필요한 기능/섹션 제거. 사용자가 필요 없다면 스킵

---

## Phase 0: 초기 컨텍스트 수집

스킬 시작 시 다음을 먼저 확인:

### 0-1. 앱소싱 리포트 확인
- `$ARGUMENTS`에 앱소싱 리포트 경로가 있으면 읽어서 사전 데이터 확보
- 없으면 사용자에게 질문: "참고할 앱소싱 리포트가 있나요?"

### 0-2. 사전 데이터 파싱 (앱소싱 리포트가 있는 경우)
앱소싱 리포트에서 자동 추출 가능한 항목:
- 키워드, 검색량, 인기도
- 경쟁사 목록 + 평점/리뷰수/월수익
- 기능 채택률 (경쟁사 기능 분석)
- 부정 리뷰 패턴 (불만 카테고리)
- CPC/CPI 데이터
- 구독 비율, 가격대

> **핵심**: 리포트에서 추출 가능한 데이터는 질문하지 않는다. AI가 모르는 것만 질문한다.

## Phase 1: 역질문 시퀀스 (The Question Flow)

다음 카테고리 순서대로 질문한다. 각 카테고리에서 **필수 질문**은 반드시 물어보고, **조건부 질문**은 이전 답변에 따라 스킵 가능하다.

### Category A: 앱 아이덴티티 (3~5 질문)

**Q-A1 (필수)**: 앱 이름
```
앱 이름을 정하셨나요?
  a) 이미 정함 → [이름 입력]
  b) 후보가 몇 개 있음 → [후보 나열]
  c) 아직 못 정함 → 제가 제안할게요
```

**Q-A2 (필수)**: 핵심 한줄 설명
```
이 앱을 한 문장으로 설명하면?
(예: "심플하고 안정적인 생리주기 달력")
```

**Q-A3 (필수)**: 타겟 유저
```
주 타겟 유저는 누구인가요?
  a) 연령대 + 성별 (예: 20~30대 여성)
  b) 직업/상황 (예: 프리랜서, 대학생)
  c) 기타 → [자유 입력]
```

**Q-A4 (조건부)**: 경쟁사 대비 차별점
```
기존 앱 대비 핵심 차별점은 무엇인가요?
  a) 더 심플하게 (기존 앱이 너무 복잡)
  b) 특정 기능이 없어서 (어떤 기능?)
  c) 가격이 너무 비싸서
  d) 기타 → [자유 입력]
```

**Q-A5 (조건부)**: 대상 국가/통화
```
주 타겟 국가는?
  a) 한국 (₩)
  b) 미국 (US$)
  c) 글로벌
  d) 기타 → [입력]
```

## Phase 2: 기획서 작성 (섹션별 점진적 검증)

모든 질문이 끝나면, 수집된 답변 + 앱소싱 데이터를 기반으로 기획서를 **섹션별로** 작성한다.

### 작성 순서 & 검증

각 섹션을 200~400자 분량으로 작성한 후 사용자에게 확인:
```
"[섹션 N] 이렇게 작성했습니다. 수정할 부분 있나요?"
  → 수정 있으면 반영 후 다음 섹션
  → 없으면 다음 섹션으로 진행
```

## Key Principles

1. **한 번에 질문 1개** — 사용자를 압도하지 않는다
2. **객관식 우선** — 열린 질문보다 선택지 있는 질문이 답하기 쉽다
3. **YAGNI 철저** — 불필요한 기능/섹션은 제거
4. **대안 탐색** — 차별점, 수익 모델 등 핵심 결정에서 2~3개 접근법 제시 후 추천
5. **점진적 검증** — 기획서를 한꺼번에 쏟아내지 않고, 섹션별로 확인
6. **유연하게** — 사용자가 원하지 않는 섹션은 스킵 가능
7. **데이터 기반** — 앱소싱 리포트가 있으면 최대한 활용하여 질문 수 최소화
8. **build-app 호환** — 산출물은 /build-app 커맨드가 파싱 가능한 형식 유지

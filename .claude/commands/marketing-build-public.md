# /marketing-build-public 스킬

## 입력
$ARGUMENTS = MVP md 파일명 (docs/ 내 배치 전제)

예시: `/marketing-build-public 1_ZenJournal_AI_Daily_Journal.md`

## 개요

MVP 기획서를 읽고 **Build in Public** 마케팅 전략을 기획한다.
쓰레드(Threads) + 인스타그램(Instagram)을 핵심 채널로, 앱 개발 과정을 공개하며 팬층을 쌓고 출시 후 오가닉 유저를 확보하는 전략이다.

---

## 실행 순서

### Step 1: MVP 파일 + 앱소싱 데이터 통합 분석

1. `docs/$ARGUMENTS` 읽기
2. `docs/build_app_context.json` 존재하면 함께 읽기
3. **앱소싱 데이터 자동 탐색** — `docs/` 및 `build_app_context.json`의 `sourcing_data` 경로에서:
   - `앱_설명_모음.md` → 경쟁사 50개 앱의 포지셔닝/기능/타겟 파악
   - `앱소싱 리포트*.md` → 부정 리뷰 패턴, 시장 데이터, 키워드 수요
   - `앱소싱_*.xlsx` → 경쟁사 정량 데이터 (월수익, 다운로드, 평점)
4. MVP에서 추출:
   - 앱 이름, 카테고리, 타겟 국가
   - 핵심 차별점 (섹션 3)
   - 기능 목록 (섹션 4)
   - 무료/유료 구분 (섹션 5, 6)
   - 브릿지 포인트 (섹션 7)
   - MoSCoW 우선순위 (섹션 8)
   - 운영 전략 (섹션 9)
   - 마케팅 단계 (섹션 11)
   - KPI (섹션 12)
   - 디자인 가이드 (섹션 15)
5. **소싱 데이터 활용 포인트** (있는 파일만 활용, 없으면 해당 항목 스킵):
   - 앱소싱 리포트 → 경쟁사 리뷰 불만 → **필러 2(문제 공감)** 콘텐츠에 실제 리뷰 원문 인용
   - 앱소싱 xlsx → 경쟁사 월수익/다운로드 → **필러 3(수치 공개)** 시 시장 대비 포지셔닝
   - 앱설명모음 → 50개 앱 기능 분석 → **기능별 콘텐츠(Step 3)** 차별점 강조 근거
   - 앱소싱 리포트 → 키워드 검색량/Trends → **해시태그 세트** 및 **커뮤니티 타겟** 최적화
   - 앱소싱 리포트 → CPC 데이터 → 유료 키워드 vs 오가닉 키워드 전략 분리
   > 소싱 데이터가 전혀 없으면 MVP md만으로 기존 방식대로 동작

```markdown
# Build in Public 마케팅 전략 — {앱 이름}

---

## 1. 계정 컨셉

### 페르소나
- **누구**: 1인 개발자 / 인디해커
- **톤앤매너**: [앱 카테고리에 맞게 설정]
  - 일기 앱 → 따뜻하고 성찰적, "매일 기록하는 개발자"
  - 생산성 앱 → 실용적이고 데이터 드리븐, "효율 중독 개발자"
  - 건강 앱 → 진정성 있고 공감적, "건강 루틴을 코딩하는 개발자"
- **언어**: [타겟 국가에 따라 — US면 영어, KR이면 한글]
- **해시태그 세트**: #buildinpublic #indiehacker #{앱카테고리} #{앱이름}

### 프로필 설정
- **Threads 프로필**:
  - 이름: {개발자명} | Building {앱이름}
  - 바이오: "Building {앱이름} in public. {한줄 앱 설명}. Day {N} 🚀"
  - 링크: 랜딩페이지 또는 대기자 폼
- **Instagram 프로필**:
  - 이름: {앱이름} | {카테고리} App
  - 바이오: "{앱 태글라인}\n🛠️ Built by @{개발자}\n📱 Coming soon → 링크"
  - 하이라이트: 개발과정 | 기능소개 | 런칭 | 후기

## 2. 콘텐츠 필러 (Content Pillars)

| # | 필러 | 비율 | 설명 | 예시 |
|---|------|------|------|------|
| 1 | **개발 과정** | 40% | 코딩, 디자인, 의사결정 과정 | "오늘 감정 트래커 UI를 3번 갈아엎었다" |
| 2 | **문제 공감** | 25% | 타겟 유저의 페인포인트 공감 | "일기 쓰고 싶은데 뭘 써야 할지 모르겠을 때" |
| 3 | **수치 공개** | 15% | 다운로드, 수익, 리텐션 등 | "Day 30: 127 다운로드, $0 수익, 배운 것 3가지" |
| 4 | **기능 소개** | 15% | 기능별 데모/스크린샷 | "AI가 일기를 분석하면 이런 결과가 나옵니다" |
| 5 | **마일스톤** | 5% | 출시, 첫 결제, 100 유저 등 | "🎉 첫 유료 구독자! $4.99 but 기분은 $4,999" |

## 4. 타임라인 콘텐츠 캘린더

### Phase A: Pre-Build (출시 2~4주 전)
| 주차 | 월 | 화 | 수 | 목 | 금 | 토 | 일 |
|------|-----|-----|-----|-----|-----|-----|-----|
| W1 | 쓰: 시작 선언 | - | 인: 경쟁사 분석 | 쓰: 기술 스택 선택 | - | 인: 디자인 목업 | - |
| W2 | 쓰: Day N 진행 | - | 인: 기능 데모 | 쓰: 문제 공감 | - | 인: 코드 비하인드 | - |

- **Threads**: 주 3~4회 (텍스트 위주, 빠른 업데이트)
- **Instagram**: 주 2~3회 (비주얼 위주, 릴스/카루셀)

### Phase B: Build & Launch (출시 전후 2주)
| 콘텐츠 | 플랫폼 | 시점 |
|--------|--------|------|
| 카운트다운 (D-7 ~ D-1) | 쓰+인 | 매일 |

| 첫 반응 공유 | 쓰 | D+1 |

| 첫 주 수치 공개 | 쓰 | D+7 |

### Phase C: Growth (출시 후 지속)
- 주 2~3회 Threads (수치 공개 + 업데이트)
- 주 1~2회 Instagram (기능 소개 + 사용자 후기)
- 마일스톤마다 특별 콘텐츠 (100 유저, 첫 수익, MRR 달성 등)

## 5. 콘텐츠 템플릿 라이브러리

### Threads 템플릿

#### 템플릿 A: 개발 일지
```
Day {N} of building {앱이름}:

{오늘 한 것 1줄 요약}

{비하인드/고민/선택 이유 2-3문장}

{질문 또는 CTA}

#buildinpublic #indiehacker #{카테고리}
```

#### 템플릿 B: 문제 공감
```
{타겟 유저의 페인포인트를 질문 형태로}

{공감 1-2문장}

{앱이 해결하는 방법 1문장 — 너무 광고스럽지 않게}

{의견 요청}
```
> **소싱 데이터 활용**: 리포트의 "부정적 리뷰 불만 패턴" 섹션에서 실제 리뷰 원문을 인용하여 Hook을 작성한다.
> 예: "I just want a simple timer, not 50 features I'll never use" (실제 경쟁사 리뷰)
> → "명상 앱에 50개 기능이 필요할까? 진짜 필요한 건 딱 하나."

#### 템플릿 B-2: 경쟁사 비교 (소싱 데이터 필수)
```
{경쟁사 앱}에서 {불만 카테고리} 때문에 별 1개 리뷰가 {N}%나 된다고?

{공감 1문장}

{우리 앱은 이걸 어떻게 해결하는지 1문장}

같은 고민 있으면 댓글로 알려주세요

#buildinpublic #{카테고리} #{앱이름}
```
> 앱소싱 xlsx 리뷰 시트의 불만 카테고리 분포 + 리포트의 경쟁사 1★ 비율 활용

#### 템플릿 C: 수치 공개
```
{앱이름} — {기간} 리포트:

📊 다운로드: {N}
💰 수익: ${N}
📈 DAU: {N}
⭐ 평점: {N}

{배운 점 1가지}

{다음 목표}
```

#### 템플릿 D: 마일스톤
```
🎉 {마일스톤 내용}!

{감정/반응 1문장}

{여기까지 오는 데 걸린 시간/과정 1-2문장}

{감사 + 다음 목표}
```

### Instagram 구도 가이드

#### 카루셀 (기능 소개)
1. 슬라이드 1: Hook 텍스트 (큰 글씨, 앱 시드 컬러 배경)
2. 슬라이드 2~4: 기능 스크린샷 + 설명 오버레이
3. 슬라이드 5: CTA ("다운로드 링크 바이오에 →")
- 사이즈: 1080x1350 (4:5)
- 폰트: {디자인 가이드의 typography와 일치}
- 컬러: {seed_color} 배경 + 흰색 텍스트

#### 릴스 (개발 과정)
1. 0-3초: Hook ("이 앱 만드는 데 {N}일 걸렸습니다")
2. 3-10초: 코딩/디자인 타임랩스
3. 10-15초: 완성된 기능 데모
4. 트렌딩 오디오 사용 권장
- 사이즈: 1080x1920 (9:16)

#### 단일 이미지 (수치 공개/마일스톤)
- 앱 시드 컬러 배경
- 큰 숫자 중앙 배치
- 하단에 앱 로고 + 이름
- 사이즈: 1080x1080 (1:1)

## 7. 대기자 확보 전략

- **랜딩페이지**: 앱 소개 + 이메일 수집 폼
  - 헤드라인: {스토어 메타데이터 title}
  - 서브헤드: {subtitle}
  - CTA: "출시 알림 받기"
  - 도구: Carrd ($19/yr) 또는 Notion + Super
- **바이오 링크**: Threads/Instagram 프로필 → 랜딩페이지
- **콘텐츠 CTA**: 매 포스트 마지막에 "바이오 링크에서 대기자 등록 →"
- **목표**: 출시 전 최소 200명 대기자
```

### Step 3: 기능별 콘텐츠 상세 생성

MVP 섹션 3의 각 차별점 기능에 대해 **최소 3개씩** 콘텐츠 스크립트를 작성:

각 기능마다:
1. Threads 텍스트 1개 (개발 과정 관점)
2. Threads 텍스트 1개 (문제 공감 관점)
3. Instagram 포스트 구도 1개 (기능 데모)

MVP 섹션 4의 Must-Have 기능에 대해 **최소 1개씩** 콘텐츠 스크립트 작성.

최종 산출물: `docs/MARKETING_BUILD_PUBLIC.md`

## 규칙

- MVP 파일에 없는 정보를 추측하지 않는다
- 광고스러운 톤 금지 — Build in Public은 **진정성**이 핵심
- 한국 타겟이면 한글, US 타겟이면 영어로 콘텐츠 작성
- 디자인 가이드(섹션 15)의 컬러/스타일을 비주얼 가이드에 반영
- 경쟁사(섹션 2) 직접 비하 금지 — "대안"으로만 포지셔닝
